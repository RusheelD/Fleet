## Findings / hypotheses
- `PromptBlockCache` originally cached by user+project only, which allowed stale memory and playbook prompt blocks to bleed across different queries and conversations for up to 60 seconds.
- Memory and playbook edits did not invalidate cached prompt blocks, so users could save an update and keep getting the previous prompt composition until the TTL expired.
- `ChatService` frequently cleared the generation flag in persistent state without publishing the matching terminal `chat.session-event`, which made chat generation appear stuck or inconsistent until fallback polling noticed.
- `SearchService` claimed to provide a case-insensitive non-Postgres fallback, but the fallback path used case-sensitive `Contains`, so in-memory tests and non-Postgres providers could behave differently from production intent.
- `McpToolSessionFactory` claimed user runtime configs override system configs by name, but the merge logic actually kept the system version and discarded the user override when names collided.
- After making prompt-cache keys query/context-aware, expired cache entries could accumulate indefinitely because the cache never evicted stale entries unless a matching key was reused or manually invalidated.
- `AgentsController` later added short-lived `OutputCache` to the executions/logs endpoints, which is the opposite of what run/log monitoring wants and can visibly suppress live updates.
- The frontend live-query safety net drifted in two ways after `d7da...`: it reused cached `live` SSE state on mount before a new stream was actually confirmed, and it started relying on a 30-second default live poll in places that used to update much faster.
- `badb260` intentionally paused background polling, but for `executions`, `logs`, and related live views that is a functional regression in the desktop/webview environment because focus/background detection is not stable enough to trust for real-time monitoring.
- The initial DbContext-conflict fix serialized many execution writes through a single scoped DbContext, which reduced EF concurrency exceptions but also created a fragile bottleneck for run/log persistence and SSE snapshot reads.
- Sub-flow heuristics were improved after `d7da...`, but there was still a gap for moderate top-level two-branch leaf splits that are meaningful orchestration candidates even without nested grandchildren.

## Implemented changes
- Made prompt-cache keys query-aware for memories and query/conversation-aware for playbooks.
- Added prompt-cache invalidation hooks in `MemoryService` and `SkillService` for create, update, and delete operations.
- Added opportunistic prompt-cache cleanup so expired entries are removed instead of accumulating indefinitely.
- Published terminal `chat.session-event` updates when generation completes, fails, or times out so SSE-driven UI state stays in sync.
- Fixed non-Postgres search fallback to behave case-insensitively.
- Fixed MCP runtime config merging so user configs genuinely override system configs by name while preserving unique entries from both sources.
- Added backend tests covering prompt-cache keying/invalidation, MCP config override behavior, case-insensitive search fallback, and terminal chat-session SSE publication.
- Removed `OutputCache` from the executions/logs APIs so the monitor can read current state instead of a three-second stale snapshot.
- Reworked the execution DB queue to use fresh DbContext instances for queued writes and fresh contexts for read-side snapshots, preserving concurrency while avoiding shared-context conflicts.
- Routed planner-generated sub-flow logging and planner-to-orchestration state transitions through the same execution DB queue so those writes no longer bypass the coordinator.
- Reset server-event connection state to `connecting` on mount instead of trusting cached `live` state from an earlier stream.
- Kept live safety polling active for chat, executions, logs, and work items instead of silently stretching or pausing it when SSE appears healthy.
- Restored background polling for the critical live queries (`work-items`, `executions`, `logs`, `chat-data`, `chat-messages`) so desktop focus changes do not freeze updates.
- Loosened orchestration coverage for moderate parallel leaf branches and added regression coverage for that sub-flow decision path.
